# Sessions API Specification

> **Module:** Sessions
>
> **Status:** Approved (Phase 1)
>
> This document is the source of truth for all Session-related backend
> APIs in TurfScore.

------------------------------------------------------------------------

# 1. Overview

The Session module is the entry point of the application.

A Session represents a single turf booking where teams, players,
matches, and statistics are managed.

All other resources in the system belong to a Session.

The backend uses the internal numeric `id` for database relationships,
while the frontend and APIs use the public `session_code`.

Example:

    Database ID : 1
    Session Code: TS4821

# 3. Business Rules

-   Every session must have a unique session code.
-   Session code is generated by the backend.
-   Session code is used in URLs and APIs.
-   Internal database IDs must never be exposed in URLs.
-   A session can contain multiple teams.
-   A session can contain multiple matches.
-   Session name is optional.
-   If no name is provided, a default name may be assigned by the
    backend.
-   Phase 1 does not require authentication.

------------------------------------------------------------------------

# 4. API Endpoints

## 4.1 Create Session

**Method**

POST

**Endpoint**

    /api/sessions

**Purpose**

Creates a new cricket session.

------------------------------------------------------------------------

## 4.2 Get Session

**Method**

GET

**Endpoint**

    /api/sessions/:sessionCode

**Purpose**

Returns session details using the public session code.

------------------------------------------------------------------------

## 4.3 Update Session

**Method**

PATCH

**Endpoint**

    /api/sessions/:sessionCode

**Purpose**

Updates editable session information such as the session name.

------------------------------------------------------------------------

## 4.4 Session Dashboard

**Method**

GET

**Endpoint**

    /api/sessions/:sessionCode/dashboard

**Purpose**

Returns dashboard data including session details, teams, players,
matches and basic statistics.

------------------------------------------------------------------------

# 6. Validation Rules

Create Session

-   Session name is optional.
-   Session code must always be unique.
-   Reject invalid payloads.

Update Session

-   Session must exist.
-   Only editable fields can be updated.

Get Session

-   Session code must exist.

Dashboard

-   Session must exist.

------------------------------------------------------------------------

# 7. Database Operations

Primary Table

-   sessions

Read-only related tables

-   teams
-   players
-   matches

The dashboard endpoint aggregates data from related tables but does not
modify them.

------------------------------------------------------------------------

# 8. Transaction Flow

## Create Session

1.  Generate unique session code.
2.  Insert session record.
3.  Return created session.

## Update Session

1.  Validate session.
2.  Update allowed fields.
3.  Return updated session.

# 11. Testing Checklist

-   Create session successfully.
-   Generated session code is unique.
-   Retrieve session by code.
-   Update session name.
-   Invalid session code returns 404.
-   Duplicate session code is handled gracefully.
-   Dashboard returns aggregated data.

------------------------------------------------------------------------

# 12. Cursor Implementation Notes

Rules for implementation:

-   Do not expose internal database IDs in APIs.
-   Always use sessionCode in routes.
-   Keep controllers thin.
-   Place business logic inside services.
-   Use repository/database layer for Drizzle queries.
-   Validate every request before database access.
-   Keep all database writes transactional where required.
-   Do not implement Team, Player or Match logic in this module.
-   Stop after completing the Sessions feature and wait for review.
